export async function getProjectsForCategory(categoryId) {
const sql = SELECT p.project_id, p.title, p.date, p.location, o.name AS organization_name FROM projects p JOIN project_categories pc ON p.project_id = pc.project_id JOIN organizations o ON p.organization_id = o.organization_id WHERE pc.category_id = $1 ORDER BY p.date;;

Coach: When you wrote this model function, what was your reasoning for using JOINs instead of separate queries?

Bryton: I used the JOINs because I wanted everything in one place. In my head, “projects for a category” already includes the project info and the organization name, so it felt cleaner to let the database pull it all together instead of doing multiple separate queries. The project_categories table is the link between everything, so once I was joining that, it just made sense to keep going. It also avoids extra round‑trips and keeps the model responsible for shaping the data before it gets to the controller.

const testErrorPage = (req, res, next) => {
const err = new Error('This is a test error');
err.status = 500;
next(err);
};

export { testErrorPage };

Coach: When you wrote this function, why did you use next(err) instead of sending a response directly?

Bryton: I used next(err) because I wanted this function to behave like a "trigger" for my global error handler, not like a full response handler on its own. The whole point was to make sure the central error middleware was working, so it made more sense to pass the error along. That way, all the formatting, logging, and status handling stay in one place.
